SECTION 01
문제 정의
1.1 현재 고객 여정 분석
Cloudbric 제품을 도입하는 중소기업 IT 담당자는 아래와 같은 단절된 여정을 경험한다.
단계 현재 경험 문제점
인지 WAF+, Managed Rules 개별 페이지 탐색 통합 정보 없음
도입 제품별 개별 가입·DNS 설정·AWS 연동 별도 진행 설정 실패 시 원인 파악 어려움
운영 WAF+ 콘솔 + AWS Console 각각 로그인 단일 현황 뷰 없음
대응 제품별 로그 교차 확인 후 각각 차단 조치 대응 지연 발생
리포트 제품별 데이터 수동 취합 자동화 없음
1.2 핵심 Pain Point
Pain Point 1 — 분산된 온보딩
WAF+와 Managed Rules는 가입 경로·설정 방식·관리 콘솔이 모두 다르다. 보안 전담 인력이 없는 SMB 담당자는 제품별
매뉴얼을 따로 읽고 따로 설정해야 하며, 초기 설정 실패 시 어느 제품 문제인지 판단조차 어렵다.
Pain Point 2 — 파편화된 현황 파악
공격이 발생했을 때 WAF+ 콘솔과 AWS Console을 오가며 로그를 교차 확인해야 한다. "지금 우리 서비스가 안전한가"라는
질문에 즉시 답할 수 있는 단일 뷰가 없다.
Pain Point 3 — 대응 지연
공격 탐지 → 원인 파악 → 차단 조치까지의 과정이 제품별로 분리되어 있어 대응 시간이 늘어난다. WAF+와 Managed
Rules가 동시에 반응하는 복합 공격 상황에서는 우선순위 판단이 어렵다.

항목 내용

SECTION 02
시장 / 경쟁사 분석
2.1 시장 배경
멀티 클라우드 전환이 가속화되면서 기업들이 사용하는 보안 도구 수가 증가하고 있다. 그에 따라 관리 복잡도가 높아지는
구조적 문제가 공통적으로 나타나고 있으며, 주요 플레이어들은 "통합 관제" 방향으로 진화하고 있다.
2.2 경쟁사 벤치마킹
항목 Cloudflare One AWS Security Hub Cloudbric (현재)
포지션 통합 Zero Trust 플랫폼 AWS 생태계 통합 관제 웹 보안 특화 SaaS
통합 관제 단일 대시보드 AWS 서비스 통합 제품별 콘솔 분리
온보딩 DNS + 가이드 위자드 AWS Console 클릭 제품별 개별 진행
알림 연동 Slack / Email / PagerDuty SNS / Slack 이메일
주요 고객 중대형 기업 AWS 사용 기업 글로벌 SMB ~ 중견
약점 비용 높음, SMB엔 과함 AWS 외 환경 미지원 제품 간 통합 없음
2.3 차별화 포인트
Cloudflare One은 강력하지만 SMB에 무겁고, AWS Security Hub는 AWS 종속이다. 그 사이 "가볍게 시작할 수
있으면서 통합 관제가 되는 웹 보안 허브" 자리가 비어 있다. Cloudbric은 이미 그 고객군(1,100개+)을 확보하고 있다.
차별점 설명
웹 보안 특화 통합 WAF+·Managed Rules·Mask 등 Cloudbric 제품군에 최적화된 통합 경험
SMB 친화적 진입 DNS 변경만으로 시작, 제품이 늘어도 관리 창구 하나
클라우드 비종속 AWS·GCP·Azure·온프레미스 무관하게 동일 허브에서 관제 가능

SECTION 03
서비스 기획서
3.1 Executive Summary
Cloudbric Security Hub — 흩어진 Cloudbric 보안 제품을 하나의 플랫폼에서 도입 · 운영 · 대응까지 처리하는 통합 보안
관제 허브
3.2 서비스 개요
항목 내용
서비스명 Cloudbric Security Hub
포지션 Cloudbric 제품군 통합 관제 플랫폼
타겟 보안 전담 인력 없는 SMB IT 담당자
핵심 가치 도입 단순화 · 현황 통합 · 대응 일원화
출시 형태 기존 Cloudbric 계정 내 허브 탭 추가 (신규 가입 불필요)
3.3 핵심 가치 3가지
One Onboarding
여러 제품을 도입할 때 하나의 플로우로 끝낸다. 제품별 매뉴얼을 따로 읽고 따로 설정하던 과정을 단일 온보딩 위자드로
통합한다.
One View
WAF+, Managed Rules, Mask의 보안 현황을 하나의 대시보드에서 실시간 확인한다. "지금 안전한가"라는 질문에 즉시
답할 수 있다.
One Response
공격 감지부터 차단 조치까지 허브 안에서 완결된다. 제품을 오가며 로그를 교차 확인하던 과정을 없앤다.
3.4 핵심 기능 정의
우선순위 기능명 한 줄 설명
P0 Unified Onboarding Wizard 도입할 제품 선택 → 단계별 설정 안내 → 완료까지 단일 플로우
P0 Unified Security Dashboard 전체 제품 보안 현황 실시간 단일 뷰

P2 Team Permission 팀원 역할 기반 접근 제어

SECTION 04
데이터 흐름 설계
4.1 온보딩 데이터 흐름
단계 액터 데이터 비고
도메인 + 제품 선택 사용자 → Hub domain, products[]
온보딩 세션 생성 Hub → Session DB session_id, steps[], status
WAF+ 설정 요청 Hub → WAF+ API domain, config 비동기
Rules 활성화 요청 Hub → AWS WAF API rule_group_id, scope 비동기
완료 응답 API → Session DB step status = done
완료 알림 Hub → 알림 서비스 email / Slack webhook
4.2 대시보드 데이터 흐름
단계 액터 데이터 비고
이벤트 수집 WAF+ / Rules / Mask → Aggregator보안 이벤트 원시 데이터 실시간
정규화 · 중복 제거 Aggregator → Event DB 정규화된 이벤트 레코드
위협 레벨 산정 Event DB 내부 처리 threat_level, summary 1분 주기
현황 조회 Dashboard API → 클라이언트 JSON 응답 Polling/SSE
임계값 초과 알림 Event DB → 알림 서비스 알림 트리거
리포트 생성 Event DB → 리포트 생성기 주간/월간 집계 데이터 스케줄

SECTION 05
API 인터페이스 정의
개발 명세가 아닌 프론트-백 간 데이터 계약 수준의 인터페이스 정의입니다.
API 1 — 온보딩 세션 생성
POST /api/v1/onboarding/sessions
Request { "domain": "example.com", "products": ["waf-plus", "managed-rules"] }
Response { "session_id": "sess_abc123", "status": "in_progress", "steps": [...], "estimated_minutes": 15 }
API 2 — 온보딩 진행 상태 조회
GET /api/v1/onboarding/sessions/{session_id}
Request (없음)
Response { "session_id", "domain", "status", "progress": 50, "steps": [...] }
API 3 — 통합 보안 현황 조회
GET /api/v1/hub/dashboard?domain=example.com.=24h
Request (없음)
Response { "threat_level", "summary": {total_blocked, total_detected}, "by_product": {...}, "top_threats":
[...], "recent_events": [...] }
API 4 — 즉시 대응 (IP 차단)
POST /api/v1/hub/actions/block-ip
Request { "ip": "203.0.113.42", "domain": "example.com", "products": ["waf-plus", "managed-rules"],
"duration_hours": 24 }
Response { "action_id", "status": "success", "applied_to": ["waf-plus", "managed-rules"], "expires_at":
"..." }
